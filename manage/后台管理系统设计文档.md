## 13. 架构设计说明

### 13.1 前后端分离

#### 13.1.1 前端分离
- **用户端前端**: `front/` - 面向普通用户的前端应用
- **管理后台前端**: `manage/` - 面向管理员的后端管理系统前端

#### 13.1.2 后端架构选择

**当前架构：统一后端服务**
- 用户端和管理后台共享同一个后端服务（`node/`）
- 通过路由前缀区分：`/api/`（用户端）和 `/api/admin/`（管理后台）
- 通过中间件进行权限隔离

**架构优势：**
1. **代码复用**：共享 services、数据库连接、工具函数
2. **数据一致性**：统一数据库，数据实时同步
3. **部署简单**：单服务部署，运维成本低
4. **开发效率**：共享业务逻辑，开发速度快

**架构优化建议：**
1. **代码组织优化**：
   ```
   node/
   ├── routes/
   │   ├── api/          # 用户端路由
   │   └── admin/        # 管理后台路由
   ├── controllers/
   │   ├── api/          # 用户端控制器
   │   └── admin/        # 管理后台控制器
   └── services/
       ├── api/          # 用户端服务
       └── admin/        # 管理后台服务
   ```

2. **安全加固**：
   - 管理后台接口添加 IP 白名单
   - 独立的认证中间件
   - 操作日志完整记录
   - 敏感操作二次验证

3. **权限隔离**：
   - 严格区分用户端和管理后台路由
   - 使用 `adminAuth` 中间件验证管理员身份
   - 管理后台接口添加额外的权限检查

**未来扩展：独立后端服务**
- 当项目规模扩大、安全要求提高时，可考虑分离为独立服务
- `api-server/` - 用户端后端服务
- `admin-server/` - 管理后台后端服务
- 通过内部 API 或消息队列进行数据同步

### 13.2 数据库设计

#### 13.2.1 统一数据库
- 用户端和管理后台共享同一个数据库
- 通过表前缀或 schema 区分（可选）
- 数据实时同步，无需数据同步机制

#### 13.2.2 数据访问控制
- 用户端：只能访问和修改自己的数据
- 管理后台：可以访问和修改所有数据
- 通过中间件和业务逻辑层进行数据权限控制

### 13.3 部署架构

#### 13.3.1 开发环境
- 用户端前端：`http://localhost:3000`
- 管理后台前端：`http://localhost:3002`
- 后端服务：`http://localhost:3001`

#### 13.3.2 生产环境
- **方案一（统一后端）**：
  - 用户端前端：CDN/静态服务器
  - 管理后台前端：独立域名或子域名（如 `admin.example.com`）
  - 后端服务：统一 API 服务（`api.example.com`）
  - 通过 Nginx 反向代理区分路由

- **方案二（分离后端）**：
  - 用户端前端：CDN/静态服务器
  - 管理后台前端：独立域名（`admin.example.com`）
  - 用户端后端：`api.example.com`
  - 管理后台后端：`admin-api.example.com`
  - 完全独立部署和扩展

### 13.4 安全架构

#### 13.4.1 认证机制
- **用户端**：JWT Token（7天有效期）
- **管理后台**：JWT Token（7天有效期）+ 管理员身份验证
- 使用不同的 Token 标识区分用户和管理员

#### 13.4.2 权限控制
- **路由级别**：通过中间件验证
- **接口级别**：通过 `adminAuth` 中间件验证管理员身份
- **数据级别**：通过业务逻辑层控制数据访问范围

#### 13.4.3 安全措施
- HTTPS 加密传输
- CORS 跨域控制
- 速率限制（Rate Limiting）
- SQL 注入防护
- XSS 攻击防护
- CSRF 防护
